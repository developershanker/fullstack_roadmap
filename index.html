REST API design
Your parallel
You consume APIs every day — axios, fetch, response parsing. Designing a REST API is just the other side of that same contract. You already know what a good API feels like from the consumer side.
Routes & methods — GET (read), POST (create), PUT/PATCH (update), DELETE. Same mental model as screen navigation, but for data.
Request/response shape — You've debugged malformed API responses at Apollo247. Now you define the contract yourself.
Status codes — 200, 201, 400, 401, 403, 404, 500. You handle these on the frontend already — now you send them.
Middleware — Like React Higher Order Components. Wraps every request for auth, logging, and error handling before it hits your route handler.
2
Databases & SQL
Your parallel
Redux store = in-memory state. A database is the state, but persisted, structured, and queryable. Your Redux selectors are essentially SQL WHERE clauses. Your reducers are INSERT/UPDATE/DELETE operations.
Tables & relations — Like TypeScript interfaces, but enforced at the storage level. Users table, Orders table, linked by IDs (foreign keys).
SQL basics — SELECT, WHERE, JOIN, GROUP BY. Think of JOIN as merging two Redux slices by a shared ID field.
Indexing — Like a book index. Without it, every query scans every row. With it, the database jumps straight to the answer.
ORMs (Prisma) — Write TypeScript objects that auto-generate SQL. Same mental model as your typed Redux state — schema first, then operations.
3
Event-driven & Kafka
Your parallel
Kafka is a distributed event emitter. Your BackHandler + useEffect subscription pattern is the same concept — something happens, listeners react. Kafka does this across services at massive scale.
Producer — Emits an event. Like dispatch(action) in Redux.
Topic — A named channel. Like a Redux slice — only consumers subscribed to this topic receive the event.
Broker — The middleman that stores and routes events. Kafka's equivalent of the Redux store — persistent and central.
Consumer — Subscribes and reacts. Like useSelector — reacts when new data arrives in its topic.
4
CDC — Change Data Capture
Your parallel
CDC watches a database for any change (insert, update, delete) and fires an event. It's exactly like useEffect with a dependency array — watching a value and reacting when it changes, but at the database level.
Maxwell — Reads MySQL's binary log (the DB's internal change diary) and publishes each change as a Kafka event automatically.
Why it matters — Lets you sync two databases without tightly coupling them. NetSuite changes → your DB gets updated automatically, no polling.
Binary log — Every DB write is logged in sequence, like a git commit history. Maxwell reads this log to detect what changed.
5
Node.js backend
Your parallel
You're already in JavaScript. You've worked on apollo-insurance-ssr with Node v22. The mental jump is smaller than you think — same language, different context (server vs app). Express is basically React Router for the backend.
Express.js — Build APIs in 10 lines of code. app.get('/path', handler) — define a route, write a handler, done.
Event loop — Node is async/non-blocking. You already know Promise chains and async/await from React Native — same patterns apply.
New packages to learn — dotenv, cors, helmet, prisma, jsonwebtoken. Same npm you already use daily.
6
System design
Your parallel
3-phase NDR architecture IS a system design answer. You can already explain it. That's exactly the level of thinking full-stack interviews test — trade-offs, decoupling, incremental migration.
Scalability — What breaks when 10x more users hit your API? Horizontal scaling (more servers) vs vertical (bigger server).
Caching (Redis) — Like memoization in React. Store expensive query results in memory so you don't recompute them every time.
Decoupling — The NDR pattern. Services should not directly depend on each other. Use Kafka, APIs, and queues as the bridges.

Project's architecture, decoded

NDR — 3 phases of decoupling.

NetSuite Dependency Reduction. A real-world, production system design you can already explain. Each phase shifts the flow further away from NetSuite as the source of truth.

Phase 1 — the problem
Everything goes through NetSuite
Before NDR, every read and write hits NetSuite's API directly. Tight coupling. Expensive. Fragile. If NetSuite goes down, you go down.
Tight coupling
NetSuite down = you down. No fallback, no cache, no resilience.
Slow reads
Every read = network API call. Always paying the latency cost.
Cost per ticket
NetSuite charges per API call. At scale, every read costs money.
Phase 2 — build your own DB, sync via Kafka
Reads from your DB. Writes still go to NetSuite.
Maxwell (CDC) watches the NetSuite DB. Every change fires a Kafka event. A consumer copies the change into your own database. Now reads no longer touch NetSuite — but writes still do.
Reads are now free
No NetSuite API call required for any read operation.
Reads are fast
Your DB, your network, no third-party latency.
Writes still coupled
Writes still go directly to NetSuite. Phase 3 fixes this.
Phase 3 — full NDR
Writes also originate from your own system.
Now writes go to your DB first. Maxwell watches your DB, fires a Kafka event, and a consumer forwards the write to NetSuite via the Wrapper API. NetSuite becomes a downstream sync target — not the source of truth anymore.
Maxwell watches your DB
Not NetSuite's DB anymore — the CDC pipeline flipped direction.
Full decoupling
NetSuite down? Your system keeps running. Events queue until it recovers.
NetSuite is downstream
It receives data from you now — not the other way around.
